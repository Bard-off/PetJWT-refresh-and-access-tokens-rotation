Для закрепления навыков на среднем уровне, я рекомендую реализовать проект **«Multi-Device Session Manager» (Диспетчер сессий на нескольких устройствах)**.

Этот проект заставит вас выйти за рамки простого выпуска токенов и научит управлять ими как реальными сущностями в базе данных.

### Суть проекта
Вы создаете API для сервиса заметок или личного кабинета, где пользователь может видеть все свои активные входы (сессии) и дистанционно «выходить» из них (например, завершить сессию на украденном телефоне).



### Что нужно реализовать (Технический стек)
* **Backend:** FastAPI (как в вашем примере).
* **Database:** PostgreSQL (для пользователей) + **Redis** (для хранения и инвалидации токенов).
* **Auth:** JWT (RS256).

### Ключевые фичи (Middle level):

1.  **Хранение метаданных сессии:**
    При логине сохраняйте в Redis не просто факт наличия токена, а объект: `user_id`, `refresh_token_id`, `User-Agent` (браузер/устройство) и `IP-адрес`.

2.  **Ротация Refresh-токенов (Refresh Token Rotation):**
    При каждом обновлении access-токена старый refresh-токен должен удаляться, а выдаваться — новый. Если кто-то попытается использовать старый refresh-токен повторно, система должна пометить сессию как скомпрометированную и разлогинить пользователя везде.

3.  **Эндпоинт `/sessions` (GET):**
    Возвращает список всех активных устройств пользователя (используя данные из Redis/DB).
    *Пример ответа:* `[{"id": 1, "device": "Chrome on Windows", "ip": "192.168.1.1", "current": true}]`.

4.  **Эндпоинт `/sessions/{id}` (DELETE):**
    Позволяет отозвать конкретный refresh-токен по его ID. После этого пользователь на том устройстве не сможет обновить access-токен и будет вынужден залогиниться заново.

5.  **Интеграция с Middleware:**
    Напишите Middleware, которое на каждый запрос проверяет не только валидность JWT (подпись), но и наличие `refresh_token_id` в "белом списке" (или отсутствие в "черном списке") в Redis.

### Почему это средний уровень?
* **Работа с состоянием (Stateful JWT):** Хотя JWT по определению stateless, управление сессиями добавляет серверный контроль.
* **Redis:** Вы научитесь работать с In-memory DB для высокопроизводительных проверок.
* **Безопасность:** Вы проработаете сценарии кражи токенов и их предотвращения.

### С чего начать?
1.  Возьмите ваш текущий код и замените `fake_db` на полноценную модель в SQLAlchemy.
2.  Добавьте поле `jti` (JWT ID) в payload ваших токенов.
3.  Настройте Docker-контейнер с Redis.
4.  Реализуйте логику: «При каждом `/refresh` старый `jti` в Redis заменяется на новый».

Этот проект даст вам понимание того, как работают системы уровня Google, где в настройках безопасности можно нажать кнопку «Выйти на всех устройствах».
